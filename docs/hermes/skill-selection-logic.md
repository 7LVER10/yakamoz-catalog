# Skill Selection Logic

## Purpose

Этот документ объясняет, какой skill выбирать при обработке входящих запросов. Он связывает skills в цепочки и определяет границы: когда skill применим, а когда нужен manual review.

---

## Current skills

| Skill | Role | Trigger | Output |
|-------|------|---------|--------|
| smb-router | Точка входа, маршрутизатор | Любое входящее сообщение от клиента | Downstream-skill + обоснование |
| lead-triage | Классификатор лидов | Новый лид / заявка | Тип, приоритет, следующий шаг |
| website-brief-intake | Сборщик брифов | Запрос на сайт / лендинг | Intake summary + пробелы |
| invoice-chase | Контроль оплат | Неоплаченный счёт / follow-up | Статус + следующий шаг |

---

## Selection rules

**Использовать `smb-router` когда:**
- Пришло входящее сообщение из Telegram / WhatsApp / email / сайт
- Нужно определить тип и направить в нужный skill

**Идти сразу в `lead-triage` когда:**
- Источник сообщения уже известен и тип очевиден (например, заявка через форму на сайте)
- smb-router уже отработал и вернул `lead-triage`

**Использовать `website-brief-intake` когда:**
- Клиент запрашивает создание сайта / лендинга
- Получен бриф с описанием задачи
- Нужно проверить полноту данных перед проектированием

**Использовать `invoice-chase` когда:**
- Есть неоплаченный счёт
- Нужно проверить статус и определить follow-up
- Приближается дата оплаты

**Использовать `manual review` когда:**
- Тип запроса не определён
- Данных мало для автоматической маршрутизации
- Запрос не входит в scope существующих skills

---

## Skill chains

### Цепочка 1: входящее сообщение
```
Вход → smb-router → lead-triage → [следующий шаг]
```
Когда: клиент написал в Telegram / WhatsApp / email

### Цепочка 2: запрос на сайт
```
Вход → smb-router → lead-triage → website-brief-intake → [проектирование]
```
Когда: "Хочу лендинг для мастера маникюра"

### Цепочка 3: вопрос по оплате
```
Вход → smb-router → invoice-chase → [следующий шаг]
```
Когда: "Оплатил счёт, но не видно"

### Цепочка 4: непонятный запрос
```
Вход → smb-router → manual review → [человек решает]
```
Когда: тип не определён, данных мало

---

## Boundaries

- Skills не заменяют AGENTS.md и agent-guidelines.md
- Один skill не должен делать всю работу — он часть цепочки
- Если данных мало — остановиться и запросить уточнение
- Если нет подходящего skill — не выдумывать, а фиксировать gap
- Skills не отправляют сообщения клиентам автоматически
- Skills не принимают бизнес-решения за человека
